QA / 테스트 엔지니어 평가 기준 (총점 100점)
1. 테스트 전략 및 설계 능력 (30점)
* 요구사항 분석 및 테스트 계획(Test Plan) 수립 능력
* 테스트 시나리오 / 케이스 / 매트릭스 설계 능력
* 기능/비기능 테스트 (성능 / 보안 / 안정성 / UI 등) 수행 능력
* 테스트 범위 설정 및 우선순위 정의 능력
2. 테스트 자동화 및 도구 활용 능력 (20점)
* 테스트 자동화 도구(Selenium, Cypress, Playwright, JMeter 등) 활용
* CI/CD 환경에서 자동화 테스트 구축 경험
* API 테스트(Postman, Swagger, K6 등) 스크립트 작성 능력
* E2E / 통합 / 회귀 테스트 자동화 경험
3. 버그 분석 및 문제 해결 능력 (20점)
* 이슈 재현 및 원인 분석 능력
* 로그/모니터링/분석 도구 활용 능력
* 문제의 Severity, Priority 판단 능력
* 근본 원인 분석(RCA) 및 재발 방지 대책 수립 능력
4. 품질 관점 제품 운영 능력 (15점)
* 품질 지표(KPI) 기반 테스트 활동 수행
* 릴리즈/배포 프로세스 이해 및 품질 관리 경험
* 장애 대응 및 긴급 패치 상황 대응 능력
5. 협업 및 커뮤니케이션 (15점)
* 요구사항/설계/개발팀과의 조율 능력
* 객관적인 근거 기반 의견 제시 능력
* 테스트 결과 보고서, 버그 리포트 작성 능력

**다른 직군 경험과의 차이점 (평가 시 주의사항)**
* 개발자 경험을 QA 역량으로 평가할 때:
  - 기능 개발 경험 ≠ 테스트 전략 수립: 개발자는 기능 구현을 의미하지만, QA는 테스트 계획, 테스트 케이스 설계, 테스트 시나리오 작성을 의미합니다.
  - 단위 테스트 작성 ≠ 통합/E2E 테스트 설계: 개발자는 자신이 작성한 코드의 단위 테스트를 의미하지만, QA는 전체 시스템 관점의 통합 테스트, E2E 테스트를 의미합니다.
  - 버그 수정 경험 ≠ 버그 분석 및 재현: 개발자는 버그를 수정하지만, QA는 버그를 발견, 재현, 분석, 우선순위 판단을 의미합니다.
* PM/기획 경험을 QA 역량으로 평가할 때:
  - 요구사항 정의 경험 ≠ 테스트 요구사항 분석: PM은 비즈니스 요구사항을 정의하지만, QA는 테스트 관점에서 요구사항을 분석하고 테스트 가능한 항목으로 변환합니다.
* 평가 시 다른 직군 경험이 QA 역량의 일부 요소에만 해당할 수 있음을 고려하고, 보수적으로 평가하세요.
